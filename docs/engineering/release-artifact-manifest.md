# Release Artifact Manifest

> **Date:** 2026-06-18
> **Status:** Format defined; first signed artifact pending certificate acquisition.
> **Scope:** Canonical artifact manifest format for every Replyline GitHub Release.

This document defines the artifact manifest format, checksum plan, and public
wording rules for Replyline releases. It is the single source of truth for what
artifacts are produced, how they are verified, and what claims can be made about them.

## Artifact Manifest Format

Every GitHub Release (tag `v*`) includes artifacts in the following structure:

### Signed Release (certificate available)

| Artifact | Type | Public? | Description |
|---|---|---|---|
| `Replyline-vX.Y.Z-windows-signed.zip` | ZIP bundle | ✅ Yes | Signed `.msi` + `.exe` bundle, Authenticode-verified |
| `Replyline-vX.Y.Z-windows-signed.zip.sha256` | Checksum | ✅ Yes | SHA256 checksum for integrity verification |
| Source code (`.zip`) | Archive | ✅ Yes | Auto-generated by GitHub |
| Source code (`.tar.gz`) | Archive | ✅ Yes | Auto-generated by GitHub |
| Release notes | Markdown | ✅ Yes | From `docs/release-notes/vX.Y.Z.md` or auto-generated |

### Unsigned Release (source/developer beta only)

| Artifact | Type | Public? | Description |
|---|---|---|---|
| Source code (`.zip`) | Archive | ✅ Yes | Auto-generated by GitHub |
| Source code (`.tar.gz`) | Archive | ✅ Yes | Auto-generated by GitHub |
| Release notes | Markdown | ✅ Yes | From `docs/release-notes/vX.Y.Z.md` or auto-generated |
| `Replyline-vX.Y.Z-windows-internal-unsigned.zip` | Workflow artifact | ❌ No | CI artifact only, 90-day retention, not attached to release |

## Checksum Plan

### Generation

SHA256 checksums are generated for every artifact regardless of signing status:

```powershell
$hash = (Get-FileHash -LiteralPath $zipPath -Algorithm SHA256).Hash.ToLowerInvariant()
"$hash  $(Split-Path -Leaf $zipPath)" | Set-Content -LiteralPath $checksumPath
```

### Format

```
<hash>  Replyline-vX.Y.Z-windows-signed.zip
```

- Checksum file name: `Replyline-vX.Y.Z-windows-signed.zip.sha256`
- Format: one line per artifact, `<hex-hash>  <filename>`
- Algorithm: SHA256 only

### Verification (user-side)

```powershell
# Download both .zip and .sha256
Get-FileHash -LiteralPath .\Replyline-vX.Y.Z-windows-signed.zip -Algorithm SHA256 |
  Select-Object -ExpandProperty Hash |
  ForEach-Object { $_.ToLowerInvariant() }
# Compare against contents of .sha256 file
```

Or with `certutil`:
```cmd
certutil -hashfile Replyline-vX.Y.Z-windows-signed.zip SHA256
```

## Artifact Naming Convention

| Version | Signed ZIP | Unsigned ZIP (CI only) |
|---|---|---|
| `v0.2.0-beta.4` | `Replyline-v0.2.0-beta.4-windows-signed.zip` | `Replyline-v0.2.0-beta.4-windows-internal-unsigned.zip` |
| `v0.2.0` | `Replyline-v0.2.0-windows-signed.zip` | `Replyline-v0.2.0-windows-internal-unsigned.zip` |

Rules:
- Always include full semver (including pre-release suffix) in artifact names.
- `-windows-signed` suffix only when Authenticode verification passed in CI.
- `-windows-internal-unsigned` suffix for all unsigned builds — never attached to release.
- Never rename artifacts after CI produces them.

## Public Wording Rules

### When signed artifacts exist

- ✅ "Windows installer available on the [Releases](https://github.com/iurii-izman/replyline/releases) page."
- ✅ "Signed with an Authenticode certificate. Verify with `Get-AuthenticodeSignature`."
- ✅ "SHA256 checksums provided for every release."
- ✅ "SmartScreen may show a warning on first install — this is normal for new applications."

### What NOT to say (ever)

- ❌ "Microsoft-approved" or "Windows-certified" — Authenticode signing ≠ Microsoft approval.
- ❌ "Virus-free guarantee" — no such claim can be made.
- ❌ "No warnings on install" — SmartScreen behavior depends on reputation, not just signing.
- ❌ "Production-ready installer" — until cross-machine smoke passes on ≥2 Windows versions.

### Current unsigned posture (source/developer beta)

- ✅ "Source/developer beta — use `git clone` + `pnpm beta:start`."
- ✅ "No unsigned artifact is presented as a public installer."
- ✅ "Build from source with `pnpm tauri build`."
- ❌ Never claim "installer available" while artifacts are unsigned.

## SmartScreen Expectation Note

### First signed release

When the first signed `.msi`/`.exe` is published:

- **Expected:** SmartScreen warning — «Windows protected your PC»
- **User action:** Click «More info» → «Run anyway»
- **Duration:** Typically 1–4 weeks of install volume before reputation builds
- **EV Certificate advantage:** May bypass SmartScreen immediately or within days
- **Standard Certificate:** Requires install volume to build reputation

### What to tell users

Include this note in every release with a signed artifact:

> **Windows SmartScreen notice:** On first install, Windows may show a SmartScreen
> warning. This is expected behavior for new applications. Click «More info» →
> «Run anyway» to continue. The warning will decrease as more users install the
> application.

### Reputation building timeline

| Week | Expected behavior |
|---|---|
| 1 | SmartScreen warning on all installs |
| 2–3 | Warning rate begins to decrease |
| 4+ | After Microsoft Defender submission, may accelerate |
| 8+ | Warning disappears for most users |

## Rollback Plan

### If signing fails in CI

| Scenario | Action |
|---|---|
| Certificate import fails | Fallback to unsigned build. Artifact stays as workflow artifact (not attached to release). Release is source-only. |
| Authenticode verification fails | Artifact marked `internal-unsigned`. Not attached to release. Investigate certificate validity, timestamp server. |
| Certificate compromised | Revoke via CA portal immediately. Rotate `WINDOWS_CERTIFICATE` + `WINDOWS_CERTIFICATE_PASSWORD` secrets. Previous signed binaries remain valid until revocation propagates. |

### If signed binary has issues post-release

1. **Do NOT delete the release** — GitHub Release is immutable in practice.
2. Create a new patch release (`vX.Y.Z+1`) with the fix.
3. Update release notes to indicate the previous signed binary is superseded.
4. If the issue is signing-specific (wrong certificate, expired), revoke and re-sign with new certificate.

### Fallback to unsigned

- Unsigned builds are always produced as a fallback (workflow artifact, 90-day retention).
- Source/developer beta posture remains valid even without signed artifacts.
- README already states unsigned artifacts are not public installers.
- GitHub Releases always include source archives regardless of signing.

## Release Artifact Checklist (per tag)

- [ ] `pnpm verify:full` — all deterministic gates green
- [ ] `cargo test` — 265 tests, 0 failures
- [ ] `pnpm test:ui` — all TS tests green
- [ ] `pnpm test:contracts` — IPC, locale, prompt, docs, copy, UI shell
- [ ] `pnpm test:public-footprint` — no secret leaks
- [ ] `pnpm release:freeze:check` — advisory review complete
- [ ] Artifact built (`pnpm tauri build`)
- [ ] Authenticode verification (if signed): `Status: Valid` on all `.msi`/`.exe`
- [ ] SHA256 checksum generated
- [ ] Signed `.zip` + `.sha256` attached to GitHub Release (if signed)
- [ ] Release notes published
- [ ] `CHANGELOG.md` updated
- [ ] No public wording violations (`pnpm copy:check`)
