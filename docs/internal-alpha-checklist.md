# Replyline Internal Alpha Checklist

> Historical note: this document is kept for internal alpha traceability.
> It is not the current stable-beta release runbook.

Операционный чеклист для команды перед подключением внешних ранних тестеров.

## 0) Цель прогона

- Проверить не только "запускается", но и usefulness в реальных сложных репликах.
- Зафиксировать blocker'ы до расширения тест-группы.
- Не уходить в feature wishlist вне core flow.

## 1) Кому подходит внутренний self-test

- Люди, у кого есть регулярные рабочие звонки с напряжёнными моментами.
- Тестеры на Windows с доступом к настройке аудио/ключей.
- Не брать в первый круг сценарии "нужен transcript/history/team workspace".

## 2) Быстрый preflight (обязательно)

Одной командой (эквивалент пунктов 1–4 ниже):

- `pnpm beta:preflight`
  (`pnpm alpha:preflight` — deprecated alias for compatibility)

Или по шагам:

1. `pnpm smoke`
2. `pnpm runtime:preflight`
3. `pnpm probe:runtime`
4. `pnpm evidence:bundle`
5. `pnpm check:slo`

Если любой шаг красный -> фиксируем blocker, дальше не идём.

## 3) Сценарии первого приоритета (usefulness)

- Эскалация по срокам: "успеваете к пятнице или нет?"
- Жёсткий pushback клиента: "почему снова сдвиг?"
- Требование решения без "воды": "кто отвечает и что делаем дальше?"

На каждый сценарий:

- сделать минимум 2-3 прогона;
- оценить `say_now` как фразу, которую реально можно произнести вслух;
- отметить, помогла ли карточка ответить быстрее и спокойнее.

## 4) Что считать blocker

- hotkey нестабилен или не срабатывает в рабочем контексте;
- захват/остановка фрагмента работает непредсказуемо;
- runtime путь не даёт карточку при валидной конфигурации;
- latency делает ответ практически бесполезным в live-моменте;
- `say_now` даёт рискованный/неуместный ответ для рабочего контекста;
- trust/legal red flag (неясно, что куда уходит; формулировки вводят в заблуждение).

## 5) Как оценивать usefulness (а не только "оно работает")

Для каждого прогона фиксировать:

- Был ли момент "я реально не знал, что ответить" (да/нет).
- Можно ли произнести `say_now` без редактирования (да/нет/частично).
- Сэкономило ли это время на формулировку (да/нет, примерно сколько секунд).
- Не ухудшило ли это ситуацию в разговоре (да/нет + почему).

Если "технически работает", но usefulness стабильно низкая -> это product blocker, а не "minor UX".

## 6) Дисциплина scope во время теста

- Не принимать запросы, уводящие в transcript/history/team workflow как "must-have для alpha".
- Не расширять продукт в meeting assistant / coach framing.
- Memory layer считать отдельным future layer, не частью текущего core MVP.

## 7) Гейт перед trusted external testers

Переход возможен только если:

- internal self-test пройден без критических blocker'ов;
- есть свежие runtime artifacts + evidence bundle;
- feedback оформлен структурно (не в формате чата "вроде норм");
- claims остаются в рамках `measured` (machine-scoped) и `pending verification`.
