# Локальная разработка с Roo Code

Этот документ объясняет, как безопасно и эффективно работать с AI-агентом Roo Code в репозитории Replyline.

> **Цель:** получить предсказуемый, проверяемый результат без повреждения архитектуры, секретов или CI.

---

## 1. Рекомендуемое использование моделей

У Roo Code нет встроенного router'а между моделями. Выбор модели — ручное действие при старте сессии или переключении задачи.

| Модель            | Когда использовать                                                  |
| ----------------- | ------------------------------------------------------------------- |
| **DeepSeek**      | Рутинные задачи, маленькие правки, документация, простые тесты.     |
| **GPT-5.3-Codex** | Сложный код, мультифайловые правки, Rust/Tauri, IPC, падение smoke. |
| **Claude Sonnet** | Архитектура, ревью решений, privacy/trust model, product wording.   |

**Важно:** Roo Code не переключает модель автоматически. Перед стартом задачи оцени её сложность и выбери модель вручную. Если в процессе стало понятно, что текущая модель не тянет — прервись и пересоздай задачу с другой моделью.

---

## 2. Когда использовать DeepSeek

DeepSeek подходит для задач, где цена ошибки низкая, а контекст небольшой.

Подходящие сценарии:

- **Маленькие UI-правки** — изменить текст кнопки, поправить отступ, переставить элемент на форме.
- **CSS** — стилизация компонента, адаптация под тёмную тему, мелкие визуальные твики.
- **Документация** — создать или обновить `.md`-файл, дополнить раздел, исправить опечатку.
- **Простые тесты** —单元-тест на существующую функцию, тест на граничное значение.
- **Небольшие bugfix'ы** — исправление конкретного воспроизводимого бага с известной причиной.

**Ограничения:** DeepSeek может терять контекст на больших diff'ах или многофайловых правках. Если задача затрагивает >3 файлов или требует глубокого понимания архитектуры — эскалируй.

---

## 3. Когда эскалировать на GPT-5.3-Codex

GPT-5.3-Codex — основная модель для сложной инженерной работы.

Эскалируй, если:

- **Rust/Tauri** — любой код в `src-tauri/`, работа с памятью, unsafe-блоки, асинхронные каналы.
- **IPC** — команды Tauri, бridge между Rust и TypeScript, сериализация/десериализация.
- **Backend/frontend contract** — изменение типов в [`src/app/model.ts`](../src/app/model.ts), добавление новой Tauri-команды, синхронизация интерфейсов.
- **Сложный баг** — баг, который не удаётся локализовать за 10 минут, stack trace в Rust, race condition.
- **Большой refactor** — изменение структуры папок, разделение модуля, миграция состояния.
- **Падение `pnpm smoke` / `pnpm verify`** — контрольная сумма не сошлась, новый баг после CI-гейта.

---

## 4. Когда использовать Claude Sonnet

Claude Sonnet — модель для решений, где важна архитектурная целостность и согласованность с продуктом.

Подходящие сценарии:

- **Архитектура** — проектирование нового модуля, выбор между подходами, определение границ ответственности.
- **Privacy/trust model** — описание data flow, оценка утечки PII, соответствие [`privacy-and-trust.md`](privacy-and-trust.md).
- **Product wording** — формулировки для пользователя, проверка на соответствие [`copy-rules.md`](copy-rules.md).
- **Сложные trade-offs** — локальное vs облачное, производительность vs читаемость, время vs качество.
- **Ревью плана перед большой реализацией** — перед тем как отдать задачу GPT-5.3-Codex, покажи план Claude Sonnet для проверки.

## 5. Safe Roo Code Workflow

Стандартный цикл работы с Roo Code в этом проекте:

```
1. READ     — прочитай релевантные файлы (модель, контроллер, тесты)
2. PLAN     — сформулируй план изменений и покажи пользователю
3. DIFF     — сделай минимальный diff, только то, что нужно по плану
4. CHECK    — запусти самый быстрый релевантный чек (tsc, тест, smoke)
5. REPORT   — напиши, что сделано, какие чеки прошли, что осталось
```

### Правила внутри workflow

| Шаг        | Что важно                                                                            |
| ---------- | ------------------------------------------------------------------------------------ |
| **READ**   | Не читай лишнего. Используй `read_file` / `search_files` точечно.                    |
| **PLAN**   | Минимальный план на 3–5 пунктов. Укажи, какие файлы будут затронуты.                 |
| **DIFF**   | Не делай unrelated changes. Одна правка — один diff.                                 |
| **CHECK**  | Запусти `pnpm smoke` для любых изменений кода. Если только docs — чек не обязателен. |
| **REPORT** | Формат: "Изменил X в [file], запустил `pnpm smoke` — passed."                        |

### Quality gates (из [`AGENTS.md`](../AGENTS.md))

- Если менялся `package.json` или `pnpm-lock.yaml` → `pnpm audit:npm`
- Если менялся `src-tauri/Cargo.toml` или Rust-зависимости → `pnpm rust:deps`
- Не утверждай, что чеки прошли, если они не были реально запущены.

### Scope and privacy guardrails for product-validation tasks

- Не расширяй product scope без явного запроса: не добавляй accounts/billing/auth/cloud DB/vector DB.
- Для tester/beta evidence задач собирай только redacted summaries; raw transcripts/reports/provider bodies запрещены.
- Не предлагай "временные" обходы privacy/trust требований ради скорости эксперимента.
- Если запрос конфликтует с `AGENTS.md` или `docs/copy-rules.md`, применяй более строгий repo policy.

## 6. Опасные действия, требующие явного подтверждения

Следующие действия **запрещено выполнять без явного подтверждения пользователем**:

| Действие                               | Почему опасно                                                                                                                                |
| -------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| **Изменение зависимостей**             | `package.json`, `pnpm-lock.yaml`, `Cargo.toml`, `Cargo.lock`                                                                                 |
| **Изменение env / secrets**            | `.env`, `.env.*`, credentials manager, API keys                                                                                              |
| **`git push`**                         | Необратимая публикация в remote                                                                                                              |
| **Релиз / build signing**              | Подпись бинарников, публикация артефактов                                                                                                    |
| **Широкие перезаписи (broad rewrite)** | Переписывание >5 файлов или >30% модуля без пошагового плана                                                                                 |
| **Удаление файлов**                    | Потеря кода, тестов, документации                                                                                                            |
| **Изменение архитектурных границ**     | Перенос логики между [`model.ts`](../src/app/model.ts), [`platform.ts`](../src/app/platform.ts), [`controller.ts`](../src/app/controller.ts) |

**Протокол:** перед таким действием Roo Code должен:

1. Написать, что собирается сделать.
2. Перечислить конкретные файлы и строки.
3. Дождаться ответа `yes / confirmed` от пользователя.
4. Только после этого выполнить действие.

## 7. Примеры хороших промптов

### Маленькая UI-правка (DeepSeek)

> В компоненте `MainSurface` замени заголовок с "Capture" на "Start capture". Никаких других изменений.

### CSS-твик (DeepSeek)

> В `App.css` поправь отступ в `.capture-button` с `16px` на `12px`. Проверь, что кнопка не съезжает при resize.

### Новая Tauri-команда (GPT-5.3-Codex)

> Добавь новую Tauri-команду `get_app_version`, которая возвращает версию из `Cargo.toml`. Нужно: зарегистрировать команду в `commands.rs`, добавить тип в `types.rs`, вызвать в `controller.ts`. После реализации запусти `pnpm smoke`.

### Баг в IPC (GPT-5.3-Codex)

> При вызове `capture-stop-and-analyze` возвращается `null` для поля `summary`. Посмотри `commands.rs` и `capture_pipeline.rs`. Найди, где теряется summary, и исправь. После фикса запусти `pnpm smoke`.

### Ревью архитектуры (Claude Sonnet)

> Я планирую добавить кэш для результатов LLM. Вот план:
>
> 1. Добавить `LlmCache` в `model.ts`
> 2. Интегрировать в `controller.ts`
> 3. Связать с настройками в `settings.rs`
>
> Проверь план на соответствие архитектурным границам и оцени риски.

### Проверка copy rules (Claude Sonnet)

> Проверь новый экран онбординга в `MainSurface.tsx` на соответствие [`copy-rules.md`](copy-rules.md). Вот текст кнопок и заголовков: [вставь текст]. Найди потенциально проблемные формулировки.

### Обновление документации (DeepSeek)

> В `docs/local-roo-workflow.md` добавь раздел про работу с тестовыми фикстурами. Используй примеры из [`fixtures/ru-work-snippets.json`](../fixtures/ru-work-snippets.json).
